# Next session — the UI/UX pass: make ACC a dashboard people *want* to live in

Status: brief for the next session. Pause point: the mission-control theme
(`src/acc/templates/styles.css`) landed on branch `feature/mission-control-theme`
and Ozzy approved the direction. This session is **not** about shipping more CSS —
it is about *using* the dashboard like a real engineer, finding everything that
still breaks the taste, and turning that into a prioritized plan.

## The one rule for this session: be the user, not the builder

Open ACC on a real repo and try to actually *do the job* it promises: control and
understand a repo's AI layer. Do not read the code first. Open the HTML, and with an
engineer's intent — "what is this repo, what can the AI do here, what changed, what do
I touch next?" — scroll, search, read, follow links, and notice every moment of
friction, confusion, or ugliness. The reference for "good" is
`docs/vision/ai-control-center-mission.html`: calm authority, source-linked, impossible
to bluff, "make the machine-readable world human-readable."

Dogfood on at least two repos, because the failure modes differ:
- **A large repo** (`/Users/ozzy-mac/Projects/code-oz`, ~1,236 files) — exercises the
  summary-only/truncation path, where the experience is currently worst.
- **A small repo** (this one, `html-dash`, or a fresh fixture) — exercises the normal,
  non-truncated path and the empty/near-empty states.

Use a browser (the live server at `127.0.0.1:5501/.claude/dashboard.html`, or open the
file). Take screenshots at each step. Keep a running log: *what I tried → what I felt →
what I wish it did.*

## The walkthrough protocol (the journey to script and screenshot)

For each repo, move through these as the user, and write down the friction at each:

1. **Land cold (first 5 seconds).** Can I tell what repo this is, what providers run
   here, and what the AI surface is — without scrolling or thinking? Is the first
   impression "I understand this" or "what am I looking at?"
2. **Orient (the Overview).** Do the cards answer "what is here, why it matters, what
   changed, what should I do?" (the vision's test). Is the bento composed, or does it
   have dead space and a broken-looking empty card?
3. **Scroll.** Does the page guide me down? Do I always know which section I'm in
   (the nav pills do **not** track scroll position today)? Is the rhythm calm or noisy?
4. **Find knowledge (search).** Search for a specific thing an engineer would hunt —
   a named rule, an MCP server, a command, a concept. How fast do I find it? Are results
   readable? On the large repo, "body search is off" — how badly does that hurt, and
   what's the fix?
5. **Read (the core promise).** Open a skill / agent / prompt / doc and try to *read its
   content*. Today most rows give a title + a link out to the raw `.md`; you cannot read
   the rendered prose, the prompt, or the rule **in** the dashboard. This is the biggest
   gap vs the vision. Decide how reading should work (inline expand? a reading pane? a
   rendered preview?).
6. **Connect (relationships).** Follow a cross-reference. Does "this rule owns that
   behavior / this command calls that script / this agent uses that MCP" actually feel
   like a map, or just a flat list?
7. **Trust.** Is it fresh? What was redacted? Any warnings? The vision wants the trust
   posture visible without being scary. Right now it's one small freshness pill.
8. **Act.** Does any view suggest the safest next move (review this rule, run this
   verification, update this stale plan)? The vision's "action handoff" is absent.

## Seed findings from this session (start here, then add to them)

The theme is in; these are the taste gaps already visible:

- **Markdown renders raw in cards and search snippets.** `**bold**`, `` `code` ``, and
  angle brackets show their markers. The Overview "Open TODOs" card and the omnibox TODO
  snippets are the worst offenders. This is the #1 "not human" problem.
- **Escaping is inconsistent.** The Overview card shows `&lt;os&gt;` (double-escaped)
  while the search snippet shows `<os>`. The "escape at build, htmlUnescape at display"
  contract is not applied uniformly to every plain-text field. Audit every place text
  reaches the DOM via `textContent`.
- **You cannot read content in place.** Docs/skills/prompts are title + path only. The
  rendered `html` exists for docs but isn't surfaced for reading; inventory items have no
  body view at all. Reading is the product — fix this.
- **Truncation guts the best repo.** The large, most-worth-navigating repo becomes
  summary-only with body search off and an empty-looking Inventory. Rethink the size
  budget UX: scoped/section-lazy rendering, a "this is summarized — here's how to see
  everything" affordance, or per-section caps instead of a global blank.
- **Weak empty states.** An eyebrow label with nothing under it (truncated Inventory)
  reads as broken. Every section needs a graceful empty/summarized message.
- **Bento composition.** Overview is three cards then dead space; design real spans so it
  feels intentional at any provider count.
- **Scanning long lists.** A flat list of 535 docs / N inventory rows has weak scent —
  consider grouping, counts, type color/iconography, collapse, and sticky group heads.
- **No scroll-spy.** Nav pills should highlight the active section as you scroll.
- **Adoption / first-run delight.** Nothing tells a newcomer how to read the dashboard or
  why it's worth their time. A tasteful one-line orientation and first-open polish would
  encourage people to actually use ACC.

## What "taste" means here (the rubric to judge against)

- **Readable before pretty.** Clean rendered prose beats raw markers. No `**`, no stray
  backticks, no `&lt;`.
- **Calm, not flashy.** Restraint, hierarchy, whitespace, one confident accent. The glow
  is ambiance, not decoration.
- **Source-linked and honest.** Every card links to a file; unknown/stale/missing are
  valid, shown states. Never invent certainty.
- **Scent and flow.** From any point I can tell where I am, what's around, and how to get
  to what I need.
- **Fast and inert.** It stays one self-contained file that opens instantly.

## Hard constraints (do not break — these are the product)

- One standalone HTML file; opens from `file://`; no server, no build, no network, no CDN
  fonts; classic inline JS only (no ES modules).
- **`textContent`-only render** — no `innerHTML`/`outerHTML`/`insertAdjacentHTML`. The CI
  guard (`tests/test_appjs_guard.py`) greps `app.js`. Any markdown rendering for cards
  must build DOM nodes safely (the existing `render_markdown_safe` path is server-side;
  client-side enrichment must stay `textContent`-based or reuse the sanitized `html`).
- Deterministic, byte-stable output; redaction-at-extraction + the tripwire stay intact.
- The redaction invariant from v1.0.1 holds: every author-controlled field is redacted at
  the source (see the `next-session-handoff` memory).
